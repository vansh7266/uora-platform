1. Executive Summary

Modern high-frequency trading depends on matching engines that process millions of orders per second with sub-millisecond p99 latency, perfect price-time priority, and provable determinism. Yet there is no standard, open, reproducible benchmark for evaluating such engines end-to-end.

UORA (Unified Orderbook Resilience Architecture) is a distributed benchmarking platform that closes this gap. A contestant uploads a matching engine in C++, Rust, Go, or Python. UORA compiles it inside a network-isolated, syscall-filtered sandbox, deploys it as a hardened container, then bombards it with a distributed bot fleet replaying real LOBSTER market data. Every order is timestamped at the proxy edge with nanosecond precision; every response is checked against a reference order book using a four-level validation hierarchy that culminates in a Graph-Edit-Distance comparison.

Results stream live through Server-Sent Events to a Next.js dashboard. An Isolation-Forest anomaly detector trained on eight benchmark features flags cheating, instrumentation gaming, and unstable behaviour the rule engine would miss. Every scored submission produces a downloadable PDF performance audit.

2. Problem Statement & Motivation

2.1 The HFT Matching Engine Problem

A matching engine is the heart of any exchange. It maintains a limit order book, matches incoming orders against resting liquidity in price-time priority, generates fills, and broadcasts public market data. In real-world venues, it must do this at single-digit microsecond latency, with thousands of concurrent participants, while never violating a single market invariant. A bug — a single mis-ordered fill, a single crossed book — can produce regulatory consequences and millions of dollars in losses.

2.2 Why Benchmarking is Hard

Production matching engines are evaluated through a combination of:

  • Synthetic load tests, which exercise throughput but rarely catch correctness regressions;
  • Conformance test suites, which catch protocol bugs but say nothing about latency under load;
  • Replay tests against captured tape, which are accurate but typically run inside proprietary harnesses with no portable score.

There is no open, reproducible, end-to-end benchmark that combines throughput, tail-latency, correctness, and anomaly detection into a single ranked score.

2.3 What UORA Provides

For Engine Authors

Upload a source file. Get a real benchmark against a known-correct reference book, real latency telemetry, and a downloadable performance audit — without operating any infrastructure.

For Evaluators

A single composite score combines throughput, correctness, tail latency, and ML-detected anomalies into a ranking that is hard to game and easy to reason about.

For Educators

A live demo mode with seeded data lets students explore the full pipeline without writing any code, then submit their own engines once they're ready.

For Operators

The whole stack is decoupled across four layers communicating only by Redis Streams and Pub/Sub. Each layer scales and replays independently.

3. System Architecture

UORA is decoupled across four horizontal layers. No layer holds in-memory state that another layer depends on. All cross-layer communication uses Redis Streams (for durable work queues) or Redis Pub/Sub (for fan-out broadcasts). Persistent metrics land in TimescaleDB.

5. Sandbox & Isolation Model

Running untrusted user code is the single most security-critical part of UORA. The platform supports two execution paths sharing the same lifecycle hooks but very different trust profiles.

5.1 Production Sandbox

The production builder (uora/sandbox/builder.py) deploys every submission as a Kubernetes Pod whose runtime class is gVisor — a user-space kernel that intercepts every syscall the contestant binary makes and emulates it in Go, never letting the binary touch the real host kernel directly. Layered on top:

  • seccomp-bpf profile (infra/security/seccomp.json) denying by default, with 312 allowed syscalls / 1026 denied.
  • FROM scratch runtime image — no shell, no libc utilities, no /etc/passwd.
  • runAsNonRoot, readOnlyRootFilesystem, automountServiceAccountToken: false.
  • NetworkPolicy denying all egress except to the benchmark fleet's pod selector.
  • CPU pinning (cpus=2) and memory cap (512 MiB) enforced by cgroups.

5.2 Local Builder (No-Docker Mode)

For local development and demo, uora/sandbox/local_builder.py bypasses Docker and runs the contestant binary as a subprocess on a bound port in the range 18100–18999. It still mirrors the production lifecycle (status transitions, build log capture, port health probe) so the dashboard and downstream workers see identical events.

Trust boundary
Local-builder mode is for trusted, local development on a developer machine. It does not apply syscall filtering or network isolation. Production deployments must use the K8s + gVisor path under infra/k8s/ and infra/security/.

6. Distributed Bot Fleet

The bot fleet is the load generator. It plays a deterministic LOBSTER-format scenario tape against the contestant engine. Each bot is an asyncio coroutine inside the benchmark worker process; production deployments fan this out across multiple worker pods using Redis Streams consumer groups.

6.1 Two-Phase Benchmark

Every benchmark runs in two distinct phases on the same engine, with different goals:

  1. Deterministic correctness pass. BotCoordinator.run_correctness_pass() replays the scenario actions in original order on a pristine book. Bot timestamps are taken from the tape, not wall-clock, so the resulting fills are deterministic and can be GED-compared against the reference. This is what the validator scores.
  2. Concurrent load pass. 25 async bots (configurable up to 10k) hammer the engine for BENCHMARK_DURATION_SEC seconds with randomly-shuffled actions. This produces real latency and throughput samples — this is what the scorer reads for the p99 and TPS terms.

7. Four-Level Correctness Validation

After each correctness pass the diff engine (uora/validator/diff_engine.py) runs the contestant's response stream through four progressively stricter validators. Failing at any level reduces correctness_rate, which is a multiplicative term in the composite score — a half-correct engine has its score halved, not just docked.

Figure 3 · Four-level correctness validation hierarchy

7.1 Graph-Edit Distance in Detail

L4 is the most novel validation step. UORA models the contestant's observable behaviour as a directed graph Gc:

  • Nodes: (order_id, status) tuples, normalised so accepted ≡ pending.
  • Edges: a fill creates a directed edge from the aggressor's node to the resting order's node, labelled with (price, quantity).

The reference book replays the same action stream and produces Gr. Similarity is computed as:

determinism = 0.60 · J(Vc, Vr) + 0.40 · J(Ec, Er)

where J(·) is Jaccard similarity. This captures both status divergence (an order ended in a different state) and fill divergence (the aggressor matched a different resting order) with a single bounded similarity score in [0, 1].

8. Composite Scoring & ML Anomaly Detection

8.1 Composite Score Formula

score = (throughput × correctness × success_rate) ÷ (p99_latency_ms + resource_penalty²)
TermSourceWhy this shape
throughputTotal successful orders ÷ benchmark windowLinear reward — every extra order matters.
correctnessL1–L4 weighted pass rateMultiplicative gate. A 50%-correct engine has its score halved.
success_rateHTTP/FIX 2xx ÷ total requestsCatches engines that drop requests under load.
p99_latency_ms99th-percentile order RTT, ns timestampsPunishes the tail. A sub-ms median with a 20 ms p99 is a worse engine.
resource_penalty²cgroup CPU & memory readings sampled during the runSquared — wasted CPU/mem is convexly penalised.

8.2 ML Anomaly Detection

The composite score alone cannot catch every form of cheating. UORA trains an Isolation Forest (uora/ml_detector/detector.py) on eight derived features from each benchmark, then runs a calibrated classifier on the live feature vector after every score.

#FeatureWhat it catches
1latency_entropyFaked / cached responses (near-constant timing)
2throughput_varianceEngines that throttle during high-difficulty windows
3error_rateBurst failures masked by averaging
4p99_to_p50_ratioAsymmetric latency profiles (slow-path leaks)
5latency_trend_slopeMemory leaks, GC drift
6pattern_correlationPre-recorded responses to known LOBSTER replay
7state_transition_gedEngine produces wrong fills (sourced from validator)
8volume_conservation_deltaPhantom or missing matched quantity

Training data is a Gaussian-centred synthetic baseline (n = 600) when fewer than 10 real reference runs are available. Contamination ratio is 0.01. The detector publishes both a continuous anomaly_score ∈ [0, 1] and a boolean is_anomaly derived from a per-feature rule layer (e.g. latency_entropy < 100,000 ns ⇒ flagged).

9. Telemetry & Real-Time Leaderboard

Every order is timestamped twice: once at the Envoy proxy edge (ingress) and once at the contestant container egress. The difference is the true round-trip latency, sampled at nanosecond resolution. Envoy writes the event to TimescaleDB through a custom HTTP filter.

13. Deployment & Operations

13.1 Local Stack (no Docker required)

# Single-command bring-up — uses Homebrew Postgres/Redis/MinIO + the Docker-free builder
$ make e2e

# Or à la carte:
$ bash scripts/local_e2e.sh           # infra + reference engine + worker
$ python3 -m uora.sandbox.local_builder
$ python3 -m uvicorn uora.submission.main:app --port 8000
$ (cd uora/leaderboard && npm run dev)

13.2 Production (Kubernetes)

  • Terraform under infra/terraform/ provisions a VPC, an EKS cluster, an RDS Aurora Postgres-compatible instance, and an S3 bucket.
  • Helm charts (planned) install the four layers as separate Deployments with HPAs keyed on Redis queue depth.
  • Envoy sidecar injection is annotated on every contestant Pod.
  • Observability: Prometheus scrapes /metrics from FastAPI; Grafana dashboards for queue lag and p99 drift.
